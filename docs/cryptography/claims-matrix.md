# Publishability Claim Matrix

Date: 2026-05-26

## Purpose

This matrix maps publication-facing claims to the repository evidence that
currently supports them, the tests or artifacts that exercise them, and the
remaining proof, audit, or implementation blockers. It is intended to keep
manuscript language conservative: implementation evidence is not a security
proof, and the current crate must not be described as a production-secure
threshold ML-DSA-65 system.

The current project status is a publishable research scaffold plus a
feature-gated hazmat ML-DSA-65 backend for experiments. It is not a
production-secure threshold ML-DSA implementation.

## Status Key

- **Implemented**: code and tests support the narrow engineering claim stated.
- **Experimentally Supported**: deterministic simulations, differential tests,
  or checked-in artifacts support the claim for the modeled experiment profile.
- **Research Scaffold**: code or documentation fixes an interface, transcript
  shape, model, or replacement boundary, but does not complete the
  cryptographic construction.
- **Open Idealized Proof Path**: documentation isolates a theorem path under an
  explicit ideal functionality assumption, but the theorem and production
  realization remain unproved.
- **Production Blocked**: a production-facing claim is intentionally blocked by
  missing proof, audit, backend, operational, or side-channel work.
- **Not Claimed**: the repository documents the topic only as an exclusion,
  caveat, or future obligation.

## Manuscript Boundary

Safe umbrella phrasing:

> The repository implements a Rust research scaffold for threshold ML-DSA-65
> protocol integration, including a feature-gated hazmat backend that exercises
> standard-verifying ML-DSA-65 artifacts, typed transcripts, VSS/interpolation
> scaffolding, proof-bearing contribution boundaries, actor simulations,
> evidence-shaping paths, fail-closed production policy gates, and reproducible
> telemetry.

Unsafe umbrella phrasing:

> The repository implements a secure, production-ready threshold ML-DSA-65
> signature scheme.

The second statement remains unsupported until the formal proof obligations in
[formal-proof-scaffold.md](formal-proof-scaffold.md), the security model in
[security-model.md](security-model.md), the VSS replacement target in
[production-vss-backend.md](production-vss-backend.md), and the contribution
proof replacement target in
[proof-bearing-contribution-boundary.md](proof-bearing-contribution-boundary.md)
are implemented, proven, audited, and externally reviewed.
